Here’s a best-practice architecture for an app-building coder IDE + live preview product like Loveable / Replit, using your model split in a clean, scalable way.

1. Core product principle
Your system should not let one model directly “do everything.”

Instead, use a controlled pipeline:

User Intent → Spec → Plan → Code Changes → Validation → Recovery → Explanation

That gives you:

better reliability

less broken code

safer file edits

easier debugging

cleaner scaling later

2. Recommended system architecture
A. Frontend
Your web app frontend should handle:

chat UI

project/file explorer

code editor

preview pane

run/build status

diff review

history / versions

logs / diagnostics

Recommended stack:

React + TypeScript

Next.js or Vite + React

Monaco Editor

WebSocket / SSE for streaming

xterm.js for terminal logs

iframe or isolated preview pane

zustand or Redux Toolkit for app state

B. API / Orchestrator backend
This is the brain of the product.

It should:

receive user prompts

classify task type

call the correct model

maintain project state

generate jobs

dispatch sandbox actions

validate results

return progress to UI

Recommended stack:

Node.js + TypeScript

Fastify or NestJS

Postgres

Redis

BullMQ / queue

object storage for snapshots/artifacts

This service should be the only place allowed to:

talk to models

mutate project state

trigger builds

approve file writes

Do not let frontend talk directly to model providers for build operations.

C. Sandbox / runtime worker
A separate isolated execution layer for:

installing packages

writing files

running dev server

linting

typechecking

testing

building

capturing preview URL/logs/errors

Recommended:

containerized workers

one isolated workspace per project/session

filesystem snapshot support

CPU/memory/time limits

network restrictions by policy

Think of this as your execution engine, not your planner.

3. Best internal agent design
Use 5 roles, even if users only see 3 buttons.

1. Conversation Agent
Model: gpt-oss-120B
Purpose: fast UX and request shaping

Responsibilities:

understand user intent

ask concise clarifying questions only when necessary

convert vague requests into structured product language

summarize what changed

Should not:

write final multi-file code directly

decide unsafe file mutations

perform architecture-heavy planning alone

2. Spec Agent
Model: gpt-oss-120B or Nemotron for harder prompts
Purpose: normalize request into structured JSON

Output example:

{
  "intent": "feature_add",
  "goal": "Add authentication and dashboard",
  "stack": {
    "frontend": "React",
    "backend": "Node/Express",
    "styling": "Tailwind"
  },
  "constraints": [
    "Preserve existing app layout",
    "Use TypeScript",
    "Do not rewrite unrelated files"
  ],
  "ui_requirements": [
    "Sidebar",
    "Login page",
    "Dashboard cards"
  ],
  "backend_requirements": [
    "JWT auth",
    "Protected routes"
  ],
  "ambiguities": [
    "No database specified"
  ]
}
This layer is critical because raw user prompts are too messy to safely build from.

3. Planner Agent
Model: Nemotron 3 Super
Purpose: turn the spec into a safe execution plan

Responsibilities:

decide whether to scaffold, patch, or refactor

identify files to read first

identify files to create/update/delete

define dependency changes

define validation steps

estimate risk

break work into ordered tasks

Example planner output:

{
  "strategy": "patch_existing_project",
  "read_files_first": [
    "package.json",
    "src/App.tsx",
    "src/routes.tsx",
    "src/components/**/*"
  ],
  "tasks": [
    {
      "id": "task_1",
      "title": "Add auth dependencies and auth utility files"
    },
    {
      "id": "task_2",
      "title": "Create login page and protected route wrapper"
    },
    {
      "id": "task_3",
      "title": "Add dashboard layout and sidebar"
    }
  ],
  "write_targets": [
    "src/pages/Login.tsx",
    "src/components/ProtectedRoute.tsx",
    "src/layouts/DashboardLayout.tsx"
  ],
  "validation": [
    "npm install",
    "tsc --noEmit",
    "npm run build"
  ],
  "risk_flags": [
    "Auth flow requires storage strategy decision"
  ]
}
This should be your source of truth before code generation starts.

4. Builder Agent
Model: Qwen3 Coder
Purpose: generate precise file edits

Responsibilities:

create or update files

follow file plan

preserve existing conventions

output structured patches

avoid touching files outside scope

Best practice:
Do not ask the model for “the whole app” in one go.

Instead, ask for:

per-task edits

explicit file diffs

full-file replacement only when necessary

strict output schema

Preferred outputs:

{
  "task_id": "task_2",
  "changes": [
    {
      "path": "src/pages/Login.tsx",
      "action": "create",
      "content": "..."
    },
    {
      "path": "src/components/ProtectedRoute.tsx",
      "action": "create",
      "content": "..."
    }
  ]
}
5. Recovery / Review Agent
Model: Nemotron first, Qwen second
Purpose: handle failures and quality review

Responsibilities:

inspect lint/type/build/runtime failures

distinguish syntax error vs system design issue

reason about repeated failures

produce correction strategy

send fix instructions to Qwen

Best practice:

Qwen handles direct code fixes

Nemotron handles root-cause reasoning when fixes loop or architecture is wrong

Never let the pipeline jump directly from:

These artifacts make the system explainable and recoverable.
